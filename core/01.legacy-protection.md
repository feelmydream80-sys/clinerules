# 01. Legacy & Stable Code Protection (CRITICAL - 절대 위반 불가)

## 핵심 원칙 (모든 작업에 최우선 적용)
**사용자가 명시적으로 요청하지 않은 코드는 보호 등급에 따라 처리한다.**

정상적으로 동작하고 있는 기존 코드를 AI가 스스로 판단해서 건드리는 행위를 철저히 금지한다.

---

## 보호 등급 (3단계)

### 🔴 절대 보호 — 어떤 경우에도 수정 금지
- `/legacy/` 폴더 전체와 그 하위 파일
- 파일명에 `_legacy`, `old_`, `deprecated_`, `stable_` 등이 포함된 파일
- `docs/legacy/` 또는 `archive/` 폴더
- 사용자가 "손대지 마", "그대로 유지해", "legacy"라고 명시한 영역

→ **읽기만 허용. 승인을 받아도 수정 불가. 새 파일에 새로 작성할 것.**

### 🟡 승인 필요 — 수정 전 반드시 사용자 확인
- 요청 범위 외 파일 (예: A 파일 수정 요청 시 B 파일에 영향이 생기는 경우)
- 공통 모듈 (`static/js/modules/common/*.js`)
- 현재 정상 동작 중인 기능과 연관된 파일

→ **아래 "예외 처리 절차" 준수**

### 🟢 자율 허용 — 보고 후 수정 가능
- 요청 파일 내 명백한 오타 (변수명 오타, 문자열 오타)
- 요청 파일 내 문법 오류 (괄호 누락 등 동작에 영향을 주는 경우)

→ **수정 후 "OO 파일에서 오타 X를 Y로 수정했습니다"라고 반드시 보고**

---

## 엄격한 금지 규칙
- "더 좋게 만들자", "현대화하자", "클린 코드로 바꾸자" 같은 판단으로 기존 코드를 건드리지 마라
- 기존 코드를 복사해서 수정하거나, 기반으로 개선하려는 시도 절대 금지  
  → 새로운 기능이나 변경이 필요하면 **완전히 새로운 파일이나 새로운 모듈**에 작성

---

## 새 작업 원칙
- 새로운 기능 추가, 버그 수정 시에도 **기존 정상 동작 코드를 최대한 건드리지 말고**, 필요한 부분만 최소한으로 수정하거나 새로 작성
- "개선"이나 "리팩토링"은 사용자가 **구체적으로 요청**한 경우에만 허용

---

## 예외 처리 절차 (🟡 승인 필요 등급에 적용)

기존 정상 동작 코드를 수정해야 할 이유가 있다고 판단되면:

1. 작업을 즉시 중단한다.
2. 다음 형식으로 사용자에게 보고한다:
   ```
   [수정 필요 감지]
   - 파일: (정확한 경로)
   - 이유: (왜 수정이 필요한지)
   - 계획: (어떻게 수정할 것인지)
   - 영향: (다른 기능에 미치는 영향)
   수정을 진행해도 될까요?
   ```
3. **사용자의 명시적인 승인** ("이 파일 수정 OK", "진행해" 등)을 받은 후에만 진행한다.
4. 승인 없이 절대 수정하지 않는다.

---

## 엣지 케이스 시나리오

### 시나리오 A: 버그 수정 중 다른 파일에 동일한 버그 발견
**상황**: `routes/api.py` 수정 중 `routes/admin.py`에도 같은 버그 발견

**올바른 처리**:
1. `api.py` 수정 완료
2. "admin.py에도 동일한 버그가 있습니다. 수정할까요?" 보고
3. 사용자 승인 후 진행

**잘못된 처리**: 승인 없이 admin.py도 함께 수정

### 시나리오 B: 공통 모듈 수정이 필요한 경우
**상황**: `dateUtils.js` 함수 하나를 수정해야 해결되는 버그

**올바른 처리**:
1. `dateUtils.js`를 사용하는 모든 파일 grep으로 조사
2. 영향 범위 사용자에게 보고
3. 승인 후 수정

이 규칙을 위반하면 정상 동작하던 기능이 망가질 위험이 매우 크다. 항상 가장 보수적으로 행동하라.
