# 대시보드 아이콘/색상 표시 기능 개발 실수 회고 및 재발 방지 가이드

## 1. 개요

이 문서는 대시보드의 'Job ID별 상세 현황' 테이블에 상태(성공/경고/실패)에 따른 아이콘과 글자색을 표시하는 기능을 개발하면서 발생했던 일련의 실수와 해결 과정을 상세히 기록한 것입니다. 이 회고를 통해 동일한 실수를 방지하고, 데이터 기반의 UI 개발 시 반드시 확인해야 할 사항들을 명확히 하는 것을 목표로 합니다.

---

## 2. 발생했던 주요 실수 내역

### 실수 1: 요구사항의 불완전한 이해 및 임의 구현

- **실수 내용:** 최초 요구사항은 "상태에 따라 아이콘과 글자색이 함께 표시"되는 것이었으나, 제 임의대로 아이콘 로직을 완전히 누락하고 글자색만 적용했습니다. 또한, 색상을 결정하는 조건 로직마저 잘못 구현했습니다.
- **발생 원인:** 요구사항을 명확히 재확인하지 않고, 제 추측에 의존하여 기능을 구현했습니다.
- **문제점:** 사용자가 의도한 핵심 기능(아이콘 표시)이 누락되었고, 반쪽짜리 기능마저 오류가 있는 상태로 1차 구현이 완료되었습니다.

### 실수 2: 제공된 데이터 명세(Key) 무시

- **실수 내용:** 사용자가 "CD101"의 성공률 성공(`sr_success`) 시 글자색이 `#a82955`라고 명확한 데이터를 제공했음에도 불구하고, 코드 수정 시 이 값을 사용하지 않고 기본값인 녹색(`#28a745`)을 하드코딩하여 사용했습니다.
- **발생 원인:** 수정에 급급한 나머지, 전달받은 구체적인 데이터 명세를 면밀히 확인하지 않고 기존 코드의 구조만 얕게 수정했습니다.
- **문제점:** 특정 데이터(CD101)에 대한 예외 케이스가 전혀 처리되지 않았고, 사용자는 피드백이 반영되지 않은 똑같은 결과물을 보게 되었습니다.

### 실수 3: 프론트엔드-백엔드 간 데이터 키(Key) 이름 불일치

- **실수 내용:** 백엔드(`Python/Flask`)에서 전달하는 데이터 객체의 키 이름(예: `cnn_sucs_wrd_colr`)과 프론트엔드(`JavaScript`)에서 사용하려는 키 이름(예: `cf_success_wd_color`)이 서로 달랐습니다. 저는 백엔드의 실제 키를 확인하지 않고, 임의의 키 이름을 추측하여 사용했습니다.
- **발생 원인:** **백엔드 API가 전달하는 실제 데이터 구조를 확인하는 절차(API 응답 로깅 등)를 생략**하고, 기존 코드나 테이블 컬럼명만 보고 키 이름을 지레짐작했습니다. 이는 데이터 기반 UI 개발에서 가장 치명적인 실수입니다.
- **문제점:** 자바스크립트는 존재하지 않는 키(`cf_success_wd_color`)에 접근하려 했고, `undefined`를 반환받았습니다. 결국 `||` (OR 연산자) 뒤에 있는 기본값(녹색)만 항상 사용되어 모든 Job ID의 성공률이 녹색으로 표시되는 오류가 발생했습니다.

### 실수 4: 존재하지 않는 아이콘 데이터 요청

- **실수 내용:** 아이콘이 사라진 문제를 해결하기 위해, 프론트엔드 코드에서 `..._icon_id_code` 라는 키를 사용하도록 수정했습니다. 하지만 이 시점에 백엔드는 이 키를 데이터에 포함하여 보내주지 않고 있었습니다.
- **발생 원인:** **실수 3의 반복**. 백엔드 수정 없이 프론트엔드 코드만 먼저 수정하며, 백엔드가 당연히 해당 키로 데이터를 보내줄 것이라고 또다시 추측했습니다. 데이터 흐름의 전체(Full-stack)를 보지 않고 눈앞의 코드만 본 것입니다.
- **문제점:** 자바스크립트는 존재하지 않는 아이콘 키에 접근했고, `undefined`를 반환받았습니다. 결국 화면에서 모든 아이콘이 사라지는 현상이 발생했습니다.

## 3. 수정 과정 및 최종 해결 방안

문제 해결을 위해 데이터가 사용자에게 표시되기까지의 전체 흐름을 역추적하여 근본 원인을 찾아 해결했습니다.

### 1단계: 데이터 소스(백엔드) 문제 해결

- **파일:** `service/dashboard_service.py`
- **수정 내용:**
    1.  `IconService`를 `DashboardService`에 주입하여 아이콘 정보를 조회할 수 있도록 의존성을 추가했습니다.
    2.  `get_summary` 함수 내부에서, 관리자 설정(`mngr_sett`)을 가져온 후, 각 설정에 포함된 아이콘 ID(예: `cnn_sucs_icon_id`)를 실제 아이콘 코드(예: `🔵`)로 변환하는 로직을 추가했습니다.
    3.  변환된 아이콘 코드를 프론트엔드에서 명확히 사용할 수 있도록 `..._icon_id_code` 라는 새로운 키에 담아 최종 데이터 객체를 구성했습니다.
- **결과:** 이 수정을 통해 프론트엔드는 이제 색상 정보와 **실제 아이콘 코드 문자열**을 함께 전달받게 되어, UI를 렌더링하는 데 필요한 모든 데이터를 갖추게 되었습니다.

### 2단계: 데이터 소비자(프론트엔드) 문제 해결

- **파일:** `static/js/modules/dashboard/dashboardTable.js`
- **수정 내용:**
    1.  `jobSetting` 객체에서 값을 가져오는 모든 키 이름을 **1단계에서 수정한 백엔드가 전달하는 정확한 키 이름**으로 모두 변경했습니다.
        -   `cf_fail_cnt` → `cnn_failr_thrs_val`
        -   `cf_success_wd_color` → `cnn_sucs_wrd_colr`
        -   `sr_success_wd_color` → `sucs_rt_sucs_wrd_colr`
        -   `sr_day_th` → `dly_sucs_rt_thrs_val`
        -   (오타 수정) `CNN_SUCS_ICON_ID_code` → `cnn_sucs_icon_id_code` 등
    2.  `createPeriodRateDisplay` 헬퍼 함수가 아이콘 코드를 인자로 받아 `<span>` 태그로 아이콘을 그리도록 수정했습니다.
    3.  Job ID를 표시하는 로직에도 아이콘을 그리는 `<span>` 태그를 추가했습니다.
- **결과:** 프론트엔드는 이제 정상적인 데이터(색상, 아이콘)를 올바른 키로 참조하게 되어, 사용자가 의도한 대로 아이콘과 글자색이 모두 정확하게 표시됩니다.

## 4. 핵심 교훈 및 재발 방지 대책

1.  **착수 전, 요구사항을 자신의 언어로 요약하고 확인받는다.**
    > "제가 이해한 최종 적용 방식은 ~입니다. 이 내용이 맞을까요?" 와 같이 코딩 시작 전에 명확한 합의점을 찾는다.

2.  **데이터 명세(Data Contract)를 최우선으로 확인한다.**
    > 백엔드와 프론트엔드 사이의 데이터는 '계약'과 같다. API 개발자에게 응답 객체의 정확한 구조(키, 데이터 타입)를 요청하거나, Postman 같은 툴로 직접 API를 호출하여 실제 응답을 눈으로 확인한다. **특히 `console.log(data)`는 가장 간단하고 확실한 검증 도구이다.**

3.  **'가정'이 아닌 '검증'을 통해 개발한다.**
    > "아마 이 키 이름일 것이다" 라는 추측은 버린다. 데이터가 어디서(DB) 시작해서 어떻게(Backend) 가공되어 최종적으로(Frontend) 소비되는지 데이터의 흐름을 먼저 파악한다. 흐름의 각 단계에서 데이터가 의도대로 전달되는지 `log`를 통해 검증하는 습관을 들인다.

4.  **문제가 발생하면, 표면이 아닌 근본 원인을 찾는다.**
    > "색이 다르게 나온다"는 현상에만 집중하지 말고, "왜 이 색이 나왔을까?"를 추적해야 한다. 데이터가 없는지, 키 이름이 다른지, 조건문이 잘못되었는지 등 근본 원인을 찾아야 연쇄적인 실수를 막을 수 있다.
