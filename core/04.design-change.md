# 04. Design Change Rules (디자인/UI 변경 작업 시 특히 엄격히 준수)

"전체 디자인 변경", "XX 디자인으로 바꿔줘", "wordcloud 디자인 적용" 등의 요청이 들어오면 **반드시** 이 규칙을 최우선으로 적용한다.

---

## 변경 규모 판단 (먼저 결정)

| 규모 | 조건 | 적용 절차 |
|------|------|----------|
| **경량** | 단일 요소 수정 (버튼 색상, 텍스트, 아이콘 1개 등) | 아래 경량 절차 |
| **표준** | 복수 요소 수정, 레이아웃 변경, 페이지 단위 변경 | 아래 표준 절차 |
| **전체** | "전체 디자인 변경", "XX 디자인으로 통일" 등 | 표준 절차 + 영향 페이지 전수 목록화 |

---

## 경량 절차 (단일 요소 수정)

1. 대상 파일(HTML/CSS) 확인
2. 수정 내용 1줄 보고 후 바로 구현
   ```
   [디자인 수정] {파일명}의 {요소} → {변경 내용} 으로 수정합니다.
   ```
3. 수정 완료 보고

> 사전 승인 불필요. 수정 후 결과 보고로 충분.

---

## 표준 절차 (복수 요소 / 레이아웃 / 페이지 단위)

### 1단계: 필수 문서 확인
- `read_file docs/ui/common/screen-domain.md` — 화면 목록 및 파일 경로 확인
- `read_file docs/core/design-change-workflow.md` — 상세 작업 흐름 확인
- 대상 페이지 분석 (HTML, CSS, JavaScript)

### 2단계: 계획 수립 및 승인 요청
다음 형식으로 사용자에게 제시 후 승인받기:
```
[디자인 변경 계획]
- 대상 파일: (경로)
- 변경 내용:
  - Before: (현재 상태)
  - After:  (변경 후 상태)
- 영향 범위: (다른 페이지/컴포넌트에 미치는 영향)
진행해도 될까요?
```

### 3단계: 구현
- 승인된 계획 범위 내에서만 수정
- 계획에 없던 변경이 필요해지면 → 즉시 중단 후 추가 승인 요청

### 4단계: 완료 보고
- 수정된 파일 목록
- 변경 내용 요약
- 확인이 필요한 URL/화면

---

## 공통 원칙

- 명시적 요청 없이 기존 디자인 변경 금지
- 다른 페이지들과의 일관성 검토 필수 (표준 절차 시)
- 계획 범위를 벗어나는 "선의의 추가 개선" 금지

---

## 실수 방지 체크리스트 (표준 절차)

- [ ] 변경 규모 판단 완료 (경량 / 표준 / 전체)
- [ ] 변경 대상 파일 전체 분석 완료
- [ ] 기존 페이지와의 일관성 검토 완료
- [ ] 구체적인 적용 계획 작성 완료
- [ ] 사용자 확인 및 승인 받음

---

## 시나리오 테스트

### 시나리오 A: 단일 요소 수정 (경량)
**상황**: "로그인 버튼 색상 파란색으로 바꿔줘"

**올바른 진행**:
1. 경량 절차 적용
2. `templates/login.html` 확인
3. "[디자인 수정] login.html의 로그인 버튼 색상 → 파란색으로 수정합니다." 보고 후 바로 구현

**잘못된 진행**: screen-domain.md 읽고 → 계획 수립 → 승인 요청 (과잉 절차)

### 시나리오 B: 페이지 단위 변경 (표준)
**상황**: "감정 분석 페이지 워드클라우드 옵션 추가해줘"

**올바른 진행**:
1. 표준 절차 적용
2. screen-domain.md에서 분석 페이지 파일 확인: `web/templates/index.html`
3. batch_processor.py에서 워드클라우드 옵션 분석
4. 변경 계획 수립 후 사용자 승인 → 구현

**검토 결과**: ✅ 규모 판단으로 불필요한 절차 제거 가능
