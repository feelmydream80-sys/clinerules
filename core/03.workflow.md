## 요청 분석 규칙 (MANDATORY - 모든 작업에 적용)

- 사용자 요청을 받으면 다음을 먼저 확인:
  1. 이 요청이 "새 기능 추가"인가 "기존 기능 수정/버그 수정"인가?
  2. 에러 메시지가 있는가? 있다면 전체 맥락 확인
  3. 기존에 동작하던 것이 안 되는 것인가, 처음 만드는 것인가?
  4. **요청 대상이 명확한가?** (어떤 파일, 어떤 폴더, 어떤 저장소인지)
- 확인 없이 코드 수정 시작 금지
- 문제를 파악하면 "이 파일들을 확인하세요"라는 구체적 안내 제공

## 실제 사례 기반 교훈 (대화에서 도출한 교훈 - 지속 추가)

- "안 돼요", "작동 안 해요", "에러나요" → 기존 기능 디버깅 우선, 새 기능 구현 금지
- "기능 라이브러리 로드 실패" 에러 발생 시: API 엔드포인트, 모델 필드, 호출 경로 먼저 확인
- **"정렬/필터링/데이터 표시" 문제: SQL 단에서만 멈추지 말고 전체 파이프라인(백엔드→정제→정렬→프론트엔드)을 검증할 것. docs/verification/pipeline-analysis.md 참조**
- **사용자가 문제 위치를特定해줬으면: 그 위치를 먼저 확인하고, 사용자가 지시한 분석 순서를 따를 것. 성급한 결론 금지**
- **사용자가 "아니다", "잘못됐다"고 하면: 즉시 중단하고 "어떤 부분이 잘못되었는지" 확인할 것. 동일한 실수를 반복하지 말 것**

## 시나리오 테스트 케이스

전체 시나리오는 [docs/verification/scenarios/01-workflow-scenarios.md](docs/verification/scenarios/01-workflow-scenarios.md) 참조

---

모든 작업은 다음 순서를 엄격히 준수한다:

1. 관련 docs/ 문서 확인 (02.documentation.md 참조)
2. **요청 의도 분석 및 확인** (00-core.md 요청 분석 규칙 참조)
   - "새 기능 추가" vs "기존 기능 수정/버그 수정" 구분
   - 에러 메시지 있는 경우: 전체 맥락 확인
   - 모호한 요청: 상세화 요청 후 진행
3. 요구사항과 기존 설계 비교 → 충돌이나 모호함이 있으면 즉시 의문 제기
4. **영향도 분석 수행** (`docs/development/impact-analysis-guide.md` 참조)
   - 변경 대상 모든 사용처 전수 검색
   - 의존성 및 영향 범위 분석
   - 롤백 계획 수립
   - 단계별 배포 계획 수립
5. 요구사항을 잘못 이해한 경우:
   - 즉시 "죄송합니다. 요청을 잘못 이해했습니다."라고 명확히 밝힘
   - 오해한 부분과 근거 문서 설명
   - 올바른 방향 제안
5. 구현 후 테스트 실행 및 구체적 결과 보고

## 지침 변경 후 의무 절차
✅ **모든 지침 문서 변경/추가 후 반드시 수행**:
1.  ✅ 시나리오 작성: 사용자 요청 및 자동 판단 작업에 대한 전체 흐름 시나리오 작성
2.  ✅ 경로 검증: 00-core 나침반에서 올바른 문서로 이동하는지 검증
3.  ✅ 명시적 요청 + 암묵적 자동 작업 케이스 모두 테스트
4.  ✅ 테스트 결과를 해당 문서 하단에 영구 기록
5.  ✅ 문제점 발견시 즉시 수정 후 재테스트

## 공통 모듈 수정 시 영향도 분석 (00-core 분류표 참조)

**적용 대상**: `static/js/modules/common/*.js` 수정 시

### 1단계: 사용처 전수 조사
```bash
# 모든 import 경로 검색
grep -r "from.*{모듈명}" --include="*.js" static/js/
# 또는
rg "from.*{모듈명}" static/js/
```

### 2단계: 상대 경로 검증
1. glob으로 실제 파일 위치 확인
2. 각 import하는 파일의 filePath 기준 상위 디렉토리 구조 파악
3. 상대 경로 계산:
   - 예: `tabs/dataDefinition/dataDefinition.js` → `modules/common/dateUtils.js`
   - 차이: `../../modules/common/dateUtils.js`

### 3단계: 검증 체크리스트
| 검증 항목 | 확인 방법 |
|-----------|----------|
| 실제 파일 존재 | glob 패턴 매칭 |
| import 경로 정확성 | filePath 기준 상대 경로 계산 |
| 모든 사용처 수정 | grep 결과 기반 전수 조사 |
| 테스트 실행 | 각 페이지 로드 확인 |

### 4단계: 결과 기록
수정 완료 후 결과를 본 문서 하단에 기록:
```
[YYYY-MM-DD] {모듈명} 수정 결과:
- 수정 파일: x
- 영향 받은 파일: x
- 테스트 결과: 성공/실패
- 문제점: (있을 경우)
```
