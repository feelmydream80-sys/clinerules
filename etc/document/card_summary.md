## 카드 요약 대시보드 데이터 미표시 문제 해결 프롬프트

### 문제 현황

현재 '카드 요약 대시보드' 페이지에 접속하면 제목만 표시되고, 실제 작업 상태를 나타내는 카드가 전혀 표시되지 않는 문제가 발생하고 있습니다. 이는 페이지의 핵심 기능이 동작하지 않는 심각한 상황입니다.

**예상 원인:**

1.  **백엔드 데이터 조회 실패:** 데이터베이스에서 작업 상태 데이터를 가져오는 과정에서 오류가 발생했거나, 조회된 데이터가 없는 경우.
2.  **백엔드 데이터 가공 오류:** 조회된 데이터를 프론트엔드로 전달하기 위해 가공하는 과정에서 데이터가 누락되거나 잘못된 형식으로 변경되는 경우.
3.  **API 통신 오류:** 프론트엔드에서 백엔드 API(`/api/card_summary`)를 호출하는 과정에서 네트워크 문제나 서버 에러가 발생하는 경우.
4.  **프론트엔드 렌더링 오류:** API로부터 데이터를 정상적으로 수신했지만, 자바스크립트 오류로 인해 화면에 카드를 정상적으로 그리지 못하는 경우.

### 해결 방향

문제의 원인을 명확히 파악하기 위해, 데이터의 흐름을 단계별로 추적하는 디버깅 전략을 사용합니다. **기존 데이터 조회 로직은 절대 변경하지 않고**, 각 단계에서 데이터가 올바르게 전달되는지 확인하기 위한 로그를 추가하여 원인을 분석하고 해결합니다.

**해결 절차:**

1.  **로그 레벨 조정 (완료):** `msys_app.py`의 로그 레벨을 `DEBUG`로 조정하여, 상세한 디버그 메시지를 확인할 수 있도록 준비합니다.
2.  **백엔드 로그 추가 (완료):** `service/card_summary_service.py`에 `current_app.logger.debug`를 사용하여 다음을 확인합니다.
    *   데이터베이스에서 조회된 원본 데이터 (`job_statuses`)
    *   프론트엔드로 전달하기 직전의 가공된 데이터 (`summary_data`)
3.  **프론트엔드 로그 추가 (완료):** `static/js/pages/card_summary.js`에 `console.log`와 `console.error`를 사용하여 다음을 확인합니다.
    *   API 호출 성공 여부 및 수신된 데이터 (`summaryData`)
    *   API 호출 실패 시 발생하는 에러 메시지
4.  **로그 분석 및 원인 파악:** 애플리케이션 재시작 후 '카드 요약 대시보드' 페이지에 접속하여, 터미널(백엔드 로그)과 브라우저 개발자 도구 콘솔(프론트엔드 로그)에 출력된 내용을 분석하여 문제의 근본 원인을 찾습니다.
5.  **문제 해결:** 파악된 원인에 따라 코드 수정을 진행합니다.
6.  **코드 원상 복구:** 문제 해결이 완료되면, 디버깅을 위해 추가했던 모든 임시 로그를 깨끗하게 제거합니다.

이러한 체계적인 접근을 통해 문제의 원인을 정확히 진단하고, 기존 기능에 영향을 주지 않으면서 안전하게 문제를 해결할 수 있습니다.

### 이슈 해결 과정 및 상세 분석

이번 '카드 요약 대시보드' 데이터 미표시 문제를 해결하는 과정에서 몇 차례의 잘못된 진단과 수정이 있었습니다. 이 기록은 최종 해결에 이르기까지의 과정과 각 단계의 원인을 상세히 분석하여 유사한 문제 발생 시 참고 자료로 활용하기 위해 작성되었습니다.

#### 1. 초기 진단 실패: 잘못된 파일 수정

*   **문제점**: 처음에는 문제의 원인을 '수집 일정' 페이지(`collection_schedule.js`)의 데이터 처리 로직 오류로 잘못 판단했습니다. 이로 인해 관련 없는 파일을 수정하여 문제 해결에 실패했습니다.
*   **원인**: 페이지 이름에 대한 오해로 인해 발생한 실수입니다. '실시간 현황'이라는 기능명을 `collection_schedule`로 잘못 유추하여 불필요한 수정을 진행했습니다.
*   **교훈**: 문제 발생 시, 관련된 정확한 파일(UI, API, 서비스 등)의 범위를 명확히 파악하는 것이 중요합니다. 개발자 도구의 네트워크 탭이나 소스 코드 검색을 통해 API 호출 경로와 관련 스크립트를 먼저 확인하는 절차가 필요합니다.

#### 2. 두 번째 진단 실패: 데이터 형태에 대한 불완전한 처리

*   **문제점**: `card_summary.js` 파일을 올바르게 찾아냈지만, 데이터가 없는 경우를 단순히 `summaryData`가 `null`이거나 `undefined`인 경우로만 한정하여 처리했습니다.
*   **원인**: API는 데이터가 없을 때 `null` 뿐만 아니라, 빈 객체(`{}`) 또는 데이터 배열이 비어있는 형태로 응답을 반환할 수 있습니다. 또한, 데이터가 존재하더라도 프론트엔드에서 화면에 표시할 유효한 상태('성공', '수집중', '실패')가 아닌 '예정' 상태의 데이터만 있는 경우가 있었습니다. 이 경우 `summaryData` 자체는 `truthy`한 값으로 평가되므로, 데이터가 없는 것으로 간주되지 않아 안내 메시지가 출력되지 않았습니다.
*   **교훈**: API 응답 데이터의 모든 가능한 형태(null, undefined, 빈 객체, 빈 배열, 특정 조건의 데이터만 있는 경우 등)를 고려하여 프론트엔드에서 방어적으로 코드를 작성해야 합니다.

#### 3. 최종 해결: 실제 렌더링 결과 기반의 조건 처리

*   **해결 방법**: 최종적으로 `static/js/pages/card_summary.js` 파일을 다음과 같이 수정하여 문제를 해결했습니다.
    1.  `cardsCreated`라는 카운터 변수를 도입하여, '성공', '수집중', '실패' 상태의 카드가 실제로 생성될 때마다 카운트를 증가시켰습니다.
    2.  API로부터 받은 데이터를 모두 처리한 후, `cardsCreated` 변수의 값이 `0`이면 (즉, 화면에 그려진 카드가 하나도 없으면) "오늘 처리된 수집 데이터가 없습니다."라는 안내 메시지를 표시하도록 로직을 변경했습니다.
*   **성공 이유**: 이 방법은 API 응답 데이터의 구조나 값에 의존하지 않고, **오직 프론트엔드에서의 실제 렌더링 결과**를 기반으로 판단하기 때문에 모든 예외 케이스를 포괄할 수 있었습니다. API가 빈 객체를 반환하든, '예정' 상태의 데이터만 반환하든, 결과적으로 화면에 표시할 카드가 없다면 안내 메시지를 확실하게 띄워주게 되어 문제를 근본적으로 해결할 수 있었습니다.

이 경험을 통해 데이터 미표시 문제 발생 시, 단순히 데이터의 존재 유무(`null` 체크)만 확인할 것이 아니라, 해당 데이터가 화면에 유의미한 정보를 표시하는 데 사용될 수 있는지를 기준으로 분기 처리를 해야 한다는 중요한 교훈을 얻었습니다.
