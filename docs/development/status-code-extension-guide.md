## 3. 듀얼 모드 설계
### 3.1 운영 방식
| 단계 | 기존 고정 컬럼 | 신규 상태 테이블 |
|------|--------------|----------------|
| **읽기** | ✅ 계속 읽음 | ✅ 있으면 우선 적용 |
| **쓰기** | ✅ 계속 기록 | ✅ 항상 동기화 기록 |
| **신규 상태** | ❌ 사용 안함 | ✅ 신규 상태만 저장 |

### 3.2 읽기 우선순위 규칙
```python
# 상태값 조회 로직
def get_status(record):
    # 1순위: 신규 상태 테이블 확인
    new_status = status_dao.get_status(record.id)
    if new_status is not None:
        return new_status
    
    # 2순위: 기존 컬럼값 반환
    return record.status
```

### 3.3 쓰기 동기화 규칙
```python
# 상태값 업데이트 로직
def update_status(record_id, status_code):
    # 1. 항상 기존 컬럼 업데이트
    schedule_dao.update_status(record_id, status_code)
    
    # 2. 항상 신규 테이블에도 동기화 업데이트
    status_dao.upsert_status(record_id, status_code)
```

## 5. 롤백 계획
### 5.1 즉시 롤백 절차
1. `status_code_extension_enabled` 설정을 `False`로 변경
2. 모든 로직이 즉시 기존 고정 컬럼만 사용하도록 전환
3. 신규 테이블은 무시되고 읽지도 쓰지도 않음
4. 기존 코드는 전혀 수정없이 그대로 동작

### 5.2 데이터 복구
- 모든 기존 상태값은 계속 원본 컬럼에 유지되고 있음
- 롤백시 데이터 손실 전혀 없음
- 신규 테이블은 아카이브 후 삭제 가능
