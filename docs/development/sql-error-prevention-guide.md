# SQL 오류 예방 가이드 (SQL Error Prevention Guide)

> ✅ **모든 DDL / SQL 작업시 database-naming-standard.md 와 함께 반드시 읽을것**
> ✅ 실제 프로젝트에서 발생했던 오류만 기반으로 작성됨
> ✅ 모든 항목은 재발 방지를 위한 강제 규칙임

---

## 1. 실제 발생 오류 케이스 (실제 사례 기반)

| No | 오류 유형 | 증상 | 원인 | 회피 방법 | 강제 규칙 |
|----|----------|------|------|-----------|----------|
| 1 | SQL 예약어 충돌 | PostgreSQL 문법 오류 발생 | `DESC` 컬럼명 사용 (SQL 예약어) | `DESCR` 사용 | ❌ 절대 `DESC` 컬럼명 사용 금지 |
| 2 | DB 타입 호환 오류 | 컬럼 타입 인식 불가 | SQLite `DATETIME` 타입 | 표준 `TIMESTAMP` 사용 | ✅ 모든 날짜 타입은 `TIMESTAMP` 통일 |
| 3 | INSERT 문법 호환 오류 | 문법 오류 발생 | SQLite `INSERT OR IGNORE` 문법 | 표준 `ON CONFLICT DO NOTHING` 사용 | ✅ 모든 중복 삽입은 표준 문법 사용 |
| 4 | 표준 약어 위반 | 명명 불일치 | `STATUS`, `CODE` 등 풀네임 사용 | 표준 약어 사전 준수 | ✅ 모든 이름은 표준 약어만 사용 |
| 5 | 기본값 누락 | 데이터 무결성 오류 | `USE_YN`, `ORD` 등 기본값 누락 | 표준 컬럼 기본값 준수 | ✅ 기본 컬럼은 항상 정해진 기본값 적용 |
| 6 | 오버엔지니어링 | 불필요한 복잡성 증가 | 과도한 매핑 테이블, 중복 컬럼 | 최소 단순 구조 우선 | ✅ 기본 마스터 테이블 구조 우선 적용 |

## 1.5 SQL 작성 전 필수 절차 (가장 최우선)
✅ **✅✅ 가장 먼저 실행할 것 ✅✅**
- [ ] **반드시 실제 DDL 파일을 먼저 열어서 실제 컬럼명 확인**
- [ ] JOIN 사용시 양쪽 테이블의 실제 연결 키 컬럼명 일치 여부 확인
- [ ] 추측으로 컬럼명 작성 절대 금지. DDL에서 직접 복사해서 사용

## 3. SQL 작성 후 최종 체크리스트 (MANDATORY)
✅ 모든 SQL 작성 후 반드시 아래 항목 전부 확인할 것

- [ ] SQL 작성 전 DDL 파일 읽고 컬럼명 확인 완료했는가?
- [ ] `DESC` 컬럼명 사용하지 않았는가?
- [ ] 모든 날짜 타입이 `TIMESTAMP` 인가?
- [ ] 표준 약어만 사용했는가?
- [ ] 모든 표준 컬럼의 기본값이 올바르게 설정되었는가?
- [ ] INSERT 문법이 표준인가?
- [ ] 불필요한 테이블이나 컬럼이 없는가?
