## Git 작업 규칙 (CRITICAL - 데이터 손실 방지)

- `git reset --hard`, `git checkout -- .`, `git clean -fd` 등 **되돌릴 수 없는 명령어**는 절대 사용자 명시적 승인 없이 실행 금지
- Git 명령어 실행 전 반드시:
  1. **어떤 저장소에서 실행되는지 확인** (root 저장소 vs 서브폴더 별도 저장소 vs 서브모듈)
  2. 현재 상태(`git status`)와 변경 가능한 파일 목록 확인
  3. 사용자에게 **"이 명령어는 [영향]을 미치며, 되돌리려면 [방법]이 필요합니다. 진행하시겠습니까?"** 확인
- 로컬 커밋이 원격에 push되지 않은 경우, `git reset --hard`는 **로컬 작업물을 영구 삭제**할 수 있음
- 서브폴더가 별도의 Git 저장소인지 확인 방법: `cd 폴더명 && git remote -v`로 원격 저장소 URL 확인
- **사용자가 "XX 폴더에서 git pull 해줘"라고 요청 시:**
  1. 해당 폴더가 별도의 Git 저장소인지 확인
  2. 해당 폴더의 remote URL이 요청 의도와 일치하는지 확인
  3. root 저장소의 remote와 혼동하지 않도록 주의

## Plan Mode에서의 정보 수집 (수정 없이 가능)

### "수행" 명령 없이도 즉시 실행 가능

**Plan Mode(읽기 전용)에서도 다음은 즉시 수행:**
- 파일/폴더 조회: `read`, `glob`, `bash ls`, `grep`
- CR ID 확인 및 번호 부여
- 의존성 분석 (import/export 추적)
- 코드 흐름 분석

**핵심 원칙**:
- ✅ **정보 수집 결과는 즉시 사용자에게 보고** (단순 정보 수집은 대기 불필요)
- ❌ **금지**: "확인 필요"라고만 하고 실제 확인 도구 사용 안 함
- ❌ **금지**: 파일 수정, git 명령, write/edit 도구 사용

**예시 (올바른 응답)**:
```
[정보 확인 결과]
- 다음 CR ID: REQ-2604-010 (001~009 존재 확인)
- 수정 대상 파일: service/dashboard_service.py (내용 확인 완료)
- 문제 원인: status_code_service 빈 리스트 반환
```

> ⚠️ **아래 조건이 충족될 때만 생성한다. 조건 미충족 시 자동 생성 금지.**

| 산출물 | 생성 조건 |
|--------|----------|
| CR 보고서 | 사용자가 `git commit` 또는 "커밋해줘"를 **명시적으로 요청**한 시점 |
| 릴리즈 노트 | CR 보고서 생성 후, 작업 유형이 **"기능 작업"(feat/fix/refactor)** 일 때 |
| 둘 다 생성 안 함 | 분석, 조회, 설명 요청 (inquiry / analysis / documentation / review) |

## 작업/커밋 확인 순서

### [작업 전] - 작업 시작 시 확인

1. **01.legacy-protection.md** - 레거시 코드 보호 확인
2. **00-core.md** - 작업 유형 분류 확인 → 올바른 절차로 안내

### [커밋 시] - 커밋 요청 시 확인

3. **CR ID 중복 체크** - `.clinerules/docs/cr/` 폴더에서 기존 ID 확인 후 다음 번호 부여
4. **기능/비기능 작업 구분**
   - 기능 작업: FP(Function Point) 기반으로 개발 일수 환산
   - 비기능 작업: 맨먼스(Man-month) 기반으로 일자 정보 입력
5. **CR 보고서 작성** - FP 또는 맨먼스 기반 공수 산정

### 작업 유형 구분

| 구분 | 조건 | 공수 환산 |
|------|------|----------|
| **기능 작업** | feat, fix, refactor | FP 기반 개발 일수 환산 |
| **비기능 작업** | inquiry, analysis, documentation, review | 맨먼스 기반 일자 정보 |

### 비기능 작업 정의

| 유형 | 설명 | 예시 |
|------|------|------|
| **inquiry** | 질문/문의 | "기본값이 뭐야?", "어떤 파일이야?" |
| **analysis** | 분석/디버깅 | "데이터가 안 떠", "에러 발생" |
| **documentation** | 문서 확인/작성 | "문서 만들어줘", "설명해줘" |
| **review** | 코드/설계 리뷰 | "코드 리뷰 해줘" |

## CR ID 생성 규칙

- 형식: `REQ-yymm-nnn`
- 예시: `REQ-2604-001` (2026년 4월 첫 번째 요청)

### CR ID 중복 방지

1. **Claude가 직접 확인**: `bash` 도구로 `.clinerules/docs/cr/` 폴더 조회
   - 명령어: `ls .clinerules/docs/cr/` 또는 `Get-ChildItem .clinerules/docs/cr/`
   - 기존 ID 확인 → 다음 번호 자동 부여 (예: 001~009 존재 → 010 부여)
   - **결과 즉시 사용자에게 보고**: "확인 결과: REQ-2604-010 사용 가능"
   - ❌ **금지**: "확인 필요"라고만 하고 실제 확인 미루기

2. 기존 기능의 버그 수정이 아니면 반드시 새 CR ID 사용
   - 예: 시간대 수정 중 메모 기능 추가 → 다른 CR ID

- **".clinerules 내려받고 싶다"** → `.clinerules` 폴더의 Git 저장소에서 pull해야 함. root 저장소(msys)를 건드리면 안 됨. Git 명령어 실행 전 반드시 대상 저장소 확인
- **`git reset --hard`로 로컬 최신 커밋이 덮어씌워짐** → `git reflog`로 이전 커밋 해시 찾아 복구 가능. 하지만 예방이 최선
