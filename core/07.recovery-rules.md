# 07. 복구 작업 규칙 (Recovery Rules)

> 이 문서는 00-core.md에서 참조합니다.

---

## 복구 범위 판단 (먼저 결정)

"복구" 또는 "롤백" 요청을 받으면 **범위를 먼저 확인**하고 실행한다.

| 요청 표현 | 범위 판단 | 처리 방식 |
|----------|----------|----------|
| "전체 롤백", "모두 복구" | 전체 | 아래 전체 복구 절차 |
| "XX 파일 복구", "이 기능만 되돌려줘" | 부분 | 아래 부분 복구 절차 |
| "복구해줘" (범위 불명확) | **확인 필요** | 범위 먼저 질문 후 진행 |

---

## 전체 복구 절차

1. `git log --oneline -10` 으로 최근 커밋 목록 확인 후 사용자에게 보고
2. 복구 목표 시점 확인 (커밋 해시 또는 태그)
3. 영향 범위 보고:
   ```
   [복구 범위]
   - 복구 시점: {커밋 해시} ({커밋 메시지})
   - 이후 변경 사항이 모두 사라집니다
   - 영향 파일: {파일 목록}
   진행해도 될까요?
   ```
4. 명시적 승인 후 실행
5. 복구 후 상태 확인 및 보고

---

## 부분 복구 절차

1. 대상 파일/기능의 현재 상태 확인
2. 복구 목표 시점 확인
3. 영향 범위 보고 (다른 기능에 미치는 영향 포함)
4. 명시적 승인 후 실행
5. 복구 후 상태 확인 및 보고

---

## 복구 작업 체크리스트

- [ ] 복구 범위 확인 (전체 vs 부분)
- [ ] 현재 상태 백업 (`git status`, `git log`)
- [ ] 영향 범위 분석 (어떤 파일이 영향을 받는가?)
- [ ] 사용자에게 영향 범위 보고
- [ ] 명시적 승인 받기
- [ ] 복구 실행
- [ ] 복구 후 상태 확인

## 엣지 케이스

### 복구 후 다시 복구가 필요한 경우
`git reflog`로 복구 이전 상태로 되돌릴 수 있음을 사용자에게 안내한다.

### 복구 대상이 이미 원격에 push된 경우
`git revert`(커밋 추가 방식)와 `git reset --hard` + force push(히스토리 재작성)의 차이를 설명하고 사용자가 선택하도록 한다.
