# 이벤트 로그 권한 문제 해결 보고서

## 1. 문제 현상

- 일반 사용자(관리자 권한이 없는 계정)에게 관리자만 봐야 하는 'System' 이벤트 로그가 노출되는 문제가 발생했습니다.

## 2. 원인 분석 과정 및 해결책

### 초기 분석 및 잘못된 접근

처음에는 백엔드에서 `job_id`가 'system'인 데이터를 제대로 필터링하지 못하는 것으로 판단했습니다.

- **1차 시도**: `service/dashboard_service.py`에서 `job_id`가 'system'인 로그를 필터링하는 로직을 추가했습니다. (대소문자 문제로 실패)
- **2차 시도**: API 호출 경로가 프론트엔드에서 잘못 지정된 것을 발견하고 수정했습니다. (`/api/con_hist_event_log` -> `/api/dashboard/event-log`)
- **3차 시도**: 정상 로그까지 사라지는 문제를 해결하기 위해 프론트엔드(`static/js/modules/dashboard/eventLog.js`)에서 `window.isAdmin`을 확인하여 로그를 제어하려고 시도했습니다.

위 접근들은 모두 근본적인 원인을 해결하지 못했습니다.

### 최종 원인 분석

문제의 핵심 원인은 **백엔드와 프론트엔드의 상호작용**에 있었습니다.

1.  **백엔드**: DB에는 `job_id`가 `null`이지만, `status`가 `LOGIN_SUCCESS` 또는 `AUTH_*`인 인증 관련 로그가 존재했습니다.
2.  **프론트엔드**: `static/js/modules/dashboard/eventLog.js` 코드는 백엔드에서 받은 데이터의 `status` 값이 인증 관련 값일 경우, **`job_id`와 상관없이 화면에 무조건 'System'이라는 글자를 강제로 출력**하고 있었습니다.

따라서 백엔드에서 `job_id`가 'system'인 로그만 필터링하는 것은 아무 소용이 없었습니다. `job_id`가 없는 인증 로그가 프론트엔드에서 'System' 로그로 둔갑하고 있었기 때문입니다.

### 최종 해결책

백엔드를 **데이터 필터링의 유일한 책임 지점(Single Source of Truth)**으로 삼아, 프론트엔드가 어떻게 동작하든 민감한 데이터가 전달되지 않도록 수정했습니다.

- **수정 파일**: `service/dashboard_service.py`
- **수정 내용**:
    - 일반 사용자(non-admin)가 이벤트 로그를 요청할 경우, 서버 단에서 아래 두 가지 조건을 **모두** 확인하여 필터링합니다.
        1. `job_id`에 'system' 문자열이 포함된 모든 로그 (대소문자 무관)
        2. `status`가 인증 관련(`AUTH_*` 또는 `LOGIN_SUCCESS`)인 모든 로그
    - 필터링 과정을 명확하게 추적할 수 있도록 상세한 디버깅 로그를 추가했습니다.

이로써 일반 사용자에게는 어떤 경우에도 시스템 관련 로그가 전달되지 않도록 조치하여 문제를 해결했습니다.
