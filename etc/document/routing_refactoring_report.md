# 라우트 구조 리팩토링 및 일관성 개선 보고서

## 1. 개요

본 문서는 `msys` 애플리케이션의 라우트 구성에 대한 분석 및 개선 작업을 기록한 보고서입니다. 코드의 일관성을 높이고 유지보수성을 향상시키기 위해 페이지 렌더링 라우트와 API 라우트를 분리하고, 블루프린트 등록 방식을 일원화하는 리팩토링을 진행했습니다.

## 2. 문제점 분석

### 2.1. 페이지 렌더링과 API 라우트의 혼재

- **현상:** `routes` 디렉토리 내 대부분의 파일(`dashboard_routes.py`, `mngr_sett_routes.py` 등)에서 페이지 렌더링을 위한 `render_template` 함수와 API 응답을 위한 `jsonify` 함수가 함께 사용되고 있었습니다.
- **문제점:**
    - **가독성 저하:** 하나의 파일에 서로 다른 두 가지 역할(UI 렌더링, 데이터 제공)의 코드가 섞여 있어 코드를 이해하기 어렵습니다.
    - **유지보수 어려움:** 기능 변경 시 UI와 API 로직을 동시에 신경 써야 하므로 수정이 복잡하고 잠재적인 버그 발생 가능성이 높습니다.
    - **확장성 부족:** 새로운 페이지나 API 엔드포인트 추가 시 일관된 구조 없이 기존 파일에 코드를 덧붙이게 되어 장기적으로 코드 구조가 악화될 수 있습니다.

### 2.2. 비일관적인 블루프린트 등록 방식

- **현상:** `msys_app.py` 파일에서 블루프린트를 등록할 때, `routes/__init__.py`의 `init_app` 함수를 통한 일괄 등록 방식과 개별 블루프린트(`collection_schedule_bp`)를 직접 임포트하여 등록하는 방식이 혼용되고 있었습니다.
- **문제점:**
    - **구조적 통일성 부족:** 블루프린트 등록 위치가 분산되어 있어 전체 라우트 구조를 파악하기 어렵습니다.
    - **혼란 야기:** 새로운 블루프린트 추가 시 어떤 방식으로 등록해야 할지 혼란을 줄 수 있으며, 이는 개발자의 실수로 이어질 수 있습니다.

## 3. 개선 방안 및 리팩토링 결과

### 3.1. 라우트 역할 분리를 위한 디렉토리 구조 개편

- **조치:** `routes` 디렉토리 하위에 `ui`와 `api` 두 개의 서브 디렉토리를 생성했습니다.
    - `routes/ui/`: 페이지 렌더링(`render_template`) 관련 라우트 파일을 위치시킵니다.
    - `routes/api/`: 순수 API(`jsonify`) 관련 라우트 파일을 위치시킵니다.

- **기대 효과:**
    - **명확한 역할 분리:** 파일의 위치만으로 해당 라우트의 역할을 명확하게 알 수 있습니다.
    - **관심사 분리(SoC):** UI 관련 변경은 `ui` 디렉토리에서, 데이터 관련 변경은 `api` 디렉토리에서 독립적으로 수행할 수 있어 작업 효율성이 높아집니다.

### 3.2. 라우트 파일 분리 및 재배치

- **조치:**
    1.  `dashboard_routes.py` 파일을 `routes/ui/dashboard_routes.py`(UI)와 `routes/api/dashboard_api.py`(API)로 분리했습니다.
    2.  `collection_schedule_routes.py` 파일을 `routes/ui/collection_schedule_routes.py`로 이동했습니다.
    3.  `api_routes.py` 파일을 `routes/api/__init__.py` 로 이동하고 이름을 변경하여 API 패키지의 일부로 만들었습니다.
    4.  기존의 `dashboard_routes.py` 파일을 삭제했습니다.

- **결과:** `dashboard`와 `collection_schedule` 기능의 라우트가 역할에 따라 명확하게 분리되어 각각 `ui`와 `api` 디렉토리에 위치하게 되었습니다.

### 3.3. 블루프린트 등록 방식 일원화

- **조치:**
    1.  `msys_app.py`에서 `collection_schedule_bp`를 직접 등록하던 코드를 제거했습니다.
    2.  모든 블루프린트(새로 분리된 `dashboard_api_bp`와 이동된 `collection_schedule_bp` 포함)가 `routes/__init__.py`의 `init_app` 함수 내에서만 등록되도록 수정했습니다.

- **결과:**
    - 이제 `msys_app.py`는 라우트 초기화를 위해 `init_routes(app)` 함수만 호출하면 되므로 코드가 간결해졌습니다.
    - 모든 블루프린트의 등록 로직이 `routes/__init__.py` 파일로 집중되어 중앙에서 관리가 용이해졌습니다.

## 4. 향후 권장 사항

- 현재 `dashboard`와 `collection_schedule` 라우트에 대해 리팩토링이 완료되었습니다. `mngr_sett_routes.py`, `analysis_routes.py` 등 다른 라우트 파일들도 본 보고서에서 제시된 구조와 패턴에 따라 점진적으로 리팩토링을 진행할 것을 권장합니다.
- 새로운 라우트 추가 시 반드시 `ui`와 `api` 역할을 구분하여 적절한 디렉토리에 파일을 생성하고, `routes/__init__.py`를 통해 블루프린트를 등록하는 규칙을 준수해야 합니다.

## 5. 결론

이번 리팩토링을 통해 라우트의 역할과 책임이 명확해졌으며, 블루프린트 등록 방식이 일원화되어 코드의 전반적인 구조적 일관성이 크게 향상되었습니다. 이는 향후 애플리케이션의 유지보수성과 확장성을 높이는 데 기여할 것입니다.
