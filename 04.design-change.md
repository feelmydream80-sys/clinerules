# 04. Design Change Rules (디자인/UI 변경 작업 시 특히 엄격히 준수)

"전체 디자인 변경", "XX 디자인으로 바꿔줘", "wordcloud 디자인 적용" 등의 요청이 들어오면 **반드시** 이 규칙을 최우선으로 적용한다.

## 디자인 변경 요청 프로세스 (사용자 → Cline)
### 1. 요청 단계 (사용자)
- **요청 내용**: 구체적인 디자인 변경 요청 (예: "대시보드 프로젝트 카드를 1행에 3개로 변경")
- **현재 상태**: 문제가 있는 현재 디자인 설명 (예: "현재 1행에 1개씩 표시되어 공간 활용도가 낮음")
- **예상 결과**: 변경 후 원하는 디자인 예시 (텍스트 기반으로 표현 가능)

### 2. 분석 단계 (Cline)
- **파일 분석**: 변경 대상 파일 (HTML, CSS, JavaScript) 분석
- **구조 파악**: 현재 디자인의 레이아웃 구조, 스타일, 컴포넌트 위치 파악
- **차이점 식별**: 기존 디자인과 요청된 디자인의 차이점 분석

### 3. 계획 단계 (Cline → 사용자)
- **변경 계획**: 구체적인 수정 방안 제시 (수정할 파일, 변경 내용, 반응형 대응)
- **예상 화면**: 수정 전/후 비교 예시 (텍스트 기반 또는 시각적 표현)
- **승인 요청**: 사용자에게 계획을 제시하고 승인을 요청

### 4. 구현 단계 (Cline → 사용자)
- **코드 수정**: 계획에 따라 실제 코드 변경
- **테스트**: 변경된 디자인의 기능 테스트
- **반응형 검사**: 다양한 화면 크기에서의 디자인 확인

### 5. 완료 단계 (Cline → 사용자)
- **결과 보고**: 변경된 디자인의 구체적인 수정 내용과 결과 보고
- **문서 업데이트**: 필요한 문서 (예: dashboard-page.md)에 변경 사항 업데이트

## 필수 단계
1. 기존 디자인 적용 파일 전체 분석 (예: wordcloud.html의 .wordcloud-options, .option-card, .cards-container 등)
2. 다른 페이지들과의 디자인 차이점 식별
3. 일관성 적용 계획 수립 (페이지별 변경 사항 구체화)
4. 계획을 사용자에게 먼저 대화창에 텍스트로 예상 화면을 제시하고 승인 받은 후 구현

## 실수 방지 체크리스트
- [ ] 변경 대상 디자인 파일 전체 분석 완료
- [ ] 기존 페이지와의 일관성 검토 완료
- [ ] 구체적인 적용 계획 작성 완료
- [ ] 사용자 확인 및 승인 받음

## 예시 요청 처리 절차
### 요청 예: 대시보드 프로젝트 카드 레이아웃 변경
**사용자 요청**:
> 대시보드의 프로젝트 카드가 현재 1행에 1개씩 표시되어 공간 활용도가 낮아요. 1행에 3개의 카드로 변경해주세요.

**Cline 분석**:
- 현재 파일: `project_management/index.html`, `project_management/static/css/modern-styles.css`
- 현재 레이아웃: `.project-list`에 각 카드가 block으로 표시
- 수정 필요: `.project-list`를 grid 레이아웃으로 변경, 각 카드의 크기 조정

**Cline 계획**:
> - CSS 변경: `.project-list`를 `grid-template-columns: repeat(3, 1fr)`로 설정
> - 반응형: 1200px 이상 3열, 768px ~ 1199px 2열, 767px 이하 1열
> - 예시 화면: [수정 전/후 비교 예시]

**사용자 승인**:
> OK, 진행해주세요.

**Cline 구현**:
> - `modern-styles.css`에서 `.project-list` 스타일 변경
> - 테스트: 브라우저에서 변경 확인
> - 결과 보고: 변경 완료

이 규칙을 위반하면 디자인 불일치가 발생하기 쉽다.
