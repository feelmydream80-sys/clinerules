# 04. Design Change Rules (디자인/UI 변경 작업 시 특히 엄격히 준수)

"전체 디자인 변경", "XX 디자인으로 바꿔줘", "wordcloud 디자인 적용" 등의 요청이 들어오면 **반드시** 이 규칙을 최우선으로 적용한다.

## 🔴 필수 읽기 (MANDATORY - 디자인 변경 전 반드시 실행)

1. **`read_file docs/ui/common/screen-domain.md`** - 화면 목록 및 파일 경로 확인
2. **`read_file docs/core/design-change-workflow.md`** - 디자인 변경 작업 경로 (상세 과정)
3. 대상 페이지 분석 (HTML, CSS, JavaScript)

**참조**: 구체적인 화면 도메인은 `docs/ui/common/screen-domain.md`, 상세 작업 흐름은 `docs/core/design-change-workflow.md` 참조

- 명시적 요청 없이 기존 디자인 변경 금지

## 핵심 원칙

1. **기존 디자인 적용 파일 전체 분석** (HTML, CSS, JavaScript)
2. **다른 페이지들과의 디자인 차이점 식별**
3. **일관성 적용 계획 수립** (페이지별 변경 사항 구체화)
4. **계획을 사용자에게 먼저 대화창에 텍스트로 예상 화면을 제시하고 승인 받은 후 구현**

## 실수 방지 체크리스트

- [ ] 변경 대상 디자인 파일 전체 분석 완료
- [ ] 기존 페이지와의 일관성 검토 완료
- [ ] 구체적인 적용 계획 작성 완료
- [ ] 사용자 확인 및 승인 받음

## 시나리오 테스트

### 시나리오: 감정 분석 페이지 워드클라우드 옵션 추가

**상황**: 사용자가 분석 페이지에 워드클라우드 옵션이 없다고 보고

**올바른 진행**:
1. 00-core.md → "공통 UI/디자인" → 04.design-change.md
2. docs/ui/common/screen-domain.md에서 분석 페이지 파일 확인: `web/templates/index.html`
3. batch_processor.py에서 사용하는 워드클라우드 옵션 분석
4. index.html에 옵션 카드 추가 계획 수립
5. 사용자에게 계획 제시 및 승인 후 구현

**검토 결과**: ✅ 04.design-change.md → screen-domain.md → 실제 파일 경로로 정확한 안내 가능
